SGB API Bridge, Siber Güvenlik Başkanlığı (eski USOM) tehdit beslemesini FortiGate, Sophos, Palo Alto, pfSense, Pi-hole ve Squid gibi cihazların doğrudan tüketebileceği düz metin (one-per-line) formatına çeviren açık kaynak köprüdür. Saatlik delta sync ile sürekli güncel, fork edip kendi GitHub Pages / GitLab / Docker / Kubernetes ortamınızda da koşturabilirsiniz.

Firewall / cihaz entegrasyonu

Aşağıdaki sekmelerden cihazınızı seçin. Tüm örneklerde varsayılan base URL https://bilsectr.github.io/sgb-api-bridge/'dir; kendi fork'unuzu / Docker / GitLab sunucunuzu kullanıyorsanız sadece host kısmını değiştirin.

FortiGate'in system external-resource (External Block List / Threat Feed) özelliği bu listeleri yerel olarak destekler — manuel "indir & yükle" döngüsüne gerek yoktur, FortiGate her refresh-rate dakikasında URL'yi kendi çekip belleğine yükler.

Adım 1 — External Threat Feed'leri tanımla (CLI)

config system external-resource
    edit "SGB-Domain"
        set type domain
        set resource "https://bilsectr.github.io/sgb-api-bridge/domain-list.txt"
        set refresh-rate 60
        set status enable
    next
    edit "SGB-IP"
        set type address
        set resource "https://bilsectr.github.io/sgb-api-bridge/ip-list.txt"
        set refresh-rate 60
        set status enable
    next
    edit "SGB-URL"
        set category 192
        set type category
        set resource "https://bilsectr.github.io/sgb-api-bridge/url-list.txt"
        set refresh-rate 60
        set status enable
    next
end

GUI'den eşdeğeri: Security Fabric → External Connectors → Create New → Threat Feeds altında Domain Name, IP Address ve URL tiplerinde üç ayrı feed oluşturun.

Adım 2 — Feed'in çekildiğini doğrula

diagnose sys external-resource entry-list
diagnose sys external-resource refresh SGB-Domain

Çıktıda last-update taze ve entries ~450.000 görüyorsanız feed başarılı.

Adım 3a — IP feed'i Firewall Policy'de kullan

Policy & Objects → Addresses → Create New → Address:

  • Type: External Block List (Threat Feed)
  • External Block List: SGB-IP
  • Name: SGB-IP-Blocklist

Sonra Policy & Objects → Firewall Policy altında yeni bir kural ekleyin: Source: all · Destination: SGB-IP-Blocklist · Action: DENY. Bu kuralı diğer allow kurallarının üstüne yerleştirin ve log etkin bırakın.

Adım 3b — Domain feed'i DNS Filter'da kullan

Security Profiles → DNS Filter → Create New:

  • Domain Filter altında Use External Domain Block Lists → SGB-Domain seç
  • Action: Block

Bu DNS profilini ilgili firewall policy'sine (genelde LAN → WAN) inspect modunda bağlayın. Böylece kullanıcı zararlı bir domain'i çözmeye çalıştığında FortiGate doğrudan DNS yanıtını bloklar.

Adım 3c — Domain feed'i Web Filter'da da kullan

DNS Filter, istemci başka bir DNS sunucusuna (8.8.8.8, DoH/DoT vb.) gittiğinde devre dışı kalır. Aynı domain listesini Web Filter'a da bağlayarak HTTP/HTTPS akışında ikinci bir savunma katmanı kurabilirsiniz — istemci hangi DNS'i kullanırsa kullansın, host header / SNI bu liste ile eşleşirse trafik kesilir.

Security Profiles → Web Filter → Create New → Static URL Filter bölümünde Block invalid URLs'in altında URL Filter tablosunu açın → Add:

  • Type: Wildcard (veya yeni sürümlerde doğrudan External Resource)
  • Action: Block
  • External Resource: SGB-Domain

CLI ile:

config webfilter profile
    edit "SGB-WebFilter"
        config web
            set blacklist enable
        end
        config url-extraction
            set status enable
        end
    next
end

# External resource referansını web filter profili icinde bağla:
config webfilter urlfilter
    edit 1
        set name "SGB-URL-Filter"
        config entries
            edit 1
                set url "SGB-Domain"
                set type wildcard
                set action block
                set status enable
            next
        end
    next
end

Sonra bu Web Filter profilini ilgili firewall policy'sinde set webfilter-profile "SGB-WebFilter" ile bağlayın ve aynı policy'ye SSL inspection profili atayın.

Adım 3d — URL feed'i Web Filter'da kullan

Aynı Web Filter profiline ikinci bir external resource olarak SGB-URL ekleyin. Tam yol (path) bazlı URL'ler için SSL/SSH Inspection profilinin deep-inspection olması gerekir; SNI bazlı yakalama için certificate-inspection de yeterli olabilir.

Adım 4 — Loglarda doğrulama

# GUI: Log & Report → Forward Traffic → Action = Deny filtresi
# CLI:
diagnose debug flow filter daddr <test-ip>
diagnose debug flow trace start 10
diagnose debug enable
İpucu: SSL deep-inspection olmadan HTTPS URL'leri yalnız SNI üzerinden yakalanır. Tam yol (path) bazlı block için deep-inspection ve kurumsal CA dağıtımı gerekir. Yalnız DNS + IP bloklaması bile pratikte zararlı erişimlerin büyük çoğunluğunu keser.

Sophos XG / SFOS, harici URL'den otomatik tazelenen FQDN host group ve IP host group objeleri destekler. Bu objeleri bir Firewall Rule'a destination olarak bağlayıp Drop edebilirsiniz.

Adım 1 — IP host group (dinamik liste)

  1. System → Hosts and Services → IP Host Group → Add
  2. Name: SGB-IP
  3. Type: IP List (URL'den)
  4. URL: https://bilsectr.github.io/sgb-api-bridge/ip-list.txt
  5. IP Format: IPv4 · Refresh interval: 60 dk
  6. Save → liste yüklendiğinde sağda satır sayısı görünür.

Adım 2 — FQDN host group (domain listesi)

  1. System → Hosts and Services → FQDN Host Group → Add
  2. Name: SGB-Domain
  3. External list URL: https://bilsectr.github.io/sgb-api-bridge/domain-list.txt
  4. Refresh interval: 60 dk

Adım 3 — URL group (web filter için)

  1. Web → URL Groups → Add
  2. Name: SGB-URL · External URL DB: https://bilsectr.github.io/sgb-api-bridge/url-list.txt
  3. Sonra Web → Policies altında bu grubu Block aksiyonu ile kurumsal web policy'sine ekleyin.

Adım 4 — Firewall Rule oluştur

Rules and Policies → Firewall Rules → Add Firewall Rule → New Firewall Rule:

  • Rule name: Block-SGB-Threats
  • Source zones: LAN · Source networks: Any
  • Destination zones: WAN
  • Destination networks: SGB-IP, SGB-Domain (her ikisini de seç)
  • Action: Drop · Log firewall traffic: On

Bu kuralı kural sıralamasında Allow kurallarının üstüne taşıyın.

Adım 5 — Doğrulama

  • Monitor & Analyze → Diagnostics → Tools → Name lookup ile listeden bir test domain'i sorgulayın.
  • Reports → Traffic → Blocked raporunda kural ismiyle hit görmelisiniz.
  • Host group sayfasındaki last update sütunu son tazeleme zamanını gösterir.
İpucu: FQDN host group ~500K kayıt için bellek/CPU tüketebilir. Küçük modellerde (XGS 87/107) domain listesini yalnızca Web Policy üzerinden (URL group) uygulayıp Firewall Rule'da sadece IP listesini kullanmak daha sağlıklıdır.

Palo Alto PAN-OS bu listeler için External Dynamic List (EDL) özelliğini sunar — IP, Domain ve URL tipinde üç ayrı EDL tanımlayıp Security/URL Filter politikalarında kullanırsınız. Cihaz her recurring aralığında kendi başına URL'yi çekip günceller.

Adım 1 — EDL objelerini oluştur (CLI)

configure

set external-list SGB-IP type ip url \
    https://bilsectr.github.io/sgb-api-bridge/ip-list.txt \
    recurring hourly

set external-list SGB-Domain type domain url \
    https://bilsectr.github.io/sgb-api-bridge/domain-list.txt \
    recurring hourly

set external-list SGB-URL type url url \
    https://bilsectr.github.io/sgb-api-bridge/url-list.txt \
    recurring hourly

commit

GUI'den eşdeğeri: Objects → External Dynamic Lists → Add. Her biri için: Type, Source (URL), Check for updates: Hourly.

Adım 2 — EDL'in çekildiğini doğrula

request system external-list refresh type ip name SGB-IP
request system external-list show type ip name SGB-IP
# Çıktıda 'Total valid entries' ~14000, 'Last refresh' taze olmalı.

GUI: Objects → External Dynamic Lists → ilgili EDL'e tıkla → List Capacities ve Test Source URL butonları.

Adım 3a — IP EDL'i Security Policy'de kullan

Policies → Security → Add:

  • Name: Block-SGB-IP
  • Source Zone: trust · Source: any
  • Destination Zone: untrust · Destination Address: SGB-IP
  • Application: any · Service: application-default
  • Action: Deny · Log at session end: ✓

Bu kuralı diğer izin kurallarının üstüne taşıyın.

Adım 3b — Domain EDL'i Anti-Spyware (DNS Sinkhole) profilinde kullan

Objects → Security Profiles → Anti-Spyware → Add → DNS Policies:

  • DNS Security veya External Dynamic Lists sekmesinde SGB-Domain'i ekle
  • Policy Action: sinkhole · Packet Capture: single-packet
  • Sinkhole IPv4: 72.5.65.111 (Palo Alto varsayılan) veya kurumsal sinkhole adresiniz

Bu Anti-Spyware profilini Security Policy'lerinizdeki Profile Settings içinde Security Profile Group veya doğrudan profile olarak bağlayın.

Adım 3c — URL EDL'i URL Filtering profilinde kullan

Objects → Security Profiles → URL Filtering → Add:

  • Categories sekmesinde aşağıda EDL listesi görünür: SGB-URL → Site Access: block, User Credential Submission: block

Bu URL Filtering profilini ilgili outbound Security kuralının Actions → Profile Setting bölümüne ekleyin. HTTPS için bir Decryption policy'si (SSL Forward Proxy) varsa tam URL bazlı; yoksa SNI bazlı block çalışır.

Adım 4 — Commit ve log doğrulama

commit description "SGB threat feeds wired into security + url-filter + anti-spyware"
  • Monitor → Logs → Traffic → Filter: (action eq deny) and (rule eq Block-SGB-IP)
  • Monitor → Logs → Threat → Sinkhole/URL Filtering log'ları
  • Monitor → Logs → URL Filtering → Block kayıtları
İpucu: PAN-OS'ta EDL kapasiteleri model bazlıdır (PA-220 düşük, PA-3200+ yüksek). request system external-list show type ip name SGB-IP ile model kapasitesinin altında olduğunuzu doğrulayın; aşılırsa fazla satır silent drop edilir.
Authentication: EDL'iniz private (GitLab/Docker) ve URL HTTPS basic auth ile korunuyorsa Client Authentication alanına kimlik bilgilerini ve gerekirse Certificate Profile tanımlayın. Public GitHub Pages için ek bir şey gerekmez.

pfSense + pfBlockerNG paketi:

  2. DNSBL → DNSBL Groups → Add → URL: https://bilsectr.github.io/sgb-api-bridge/domain-list.txt · Format: auto
  3. Update sekmesinde "Force Reload" → IP & DNSBL.

Squid proxy'de saatlik cron + ACL:

acl sgb_blacklist dstdomain "/etc/squid/sgb-domain-list.txt"

# crontab -e

  -o /etc/squid/sgb-domain-list.txt && systemctl reload squid

MikroTik RouterOS 7+ için /ip/firewall/address-list import scripti:

/system script add name=sgb-sync source={
  /tool fetch url="https://bilsectr.github.io/sgb-api-bridge/ip-list.txt" \
    mode=https dst-path=sgb-ip.txt
  /ip firewall address-list remove [find list=SGB]
  :local data [/file get sgb-ip.txt contents]
  :foreach line in=[:toarray $data] do={
    :if ([:len $line] > 0) do={
      /ip firewall address-list add list=SGB address=$line
    }
  }
}

/system scheduler add name=sgb-hourly interval=1h on-event=sgb-sync

Sonra /ip firewall filter'da src-address-list=SGB / dst-address-list=SGB ile drop kuralı ekleyin.

Nasıl çalışır?

İki tür senkron mekanizması var; sürekli çalışan yalnızca delta'dır.

MekanizmaSıklıkSüreAçıklama
Delta sync Saatte 1, otomatik ~1–3 dk SGB API'den her tür için yalnız yeni kayıtları çeker. Sürekli çalışan tek mekanizma budur.
Full sync Yalnız ilk kurulum (elle) ~10–15+ sa Sıfırdan tam re-sync. Geçmiş veri zaten repo'da geldiği için fork edenler için gerekmez.
Geçmiş veri zaten repo'da. Bu repo'yu klonlayan / fork eden herkes docs/*-list.txt ve state/seen_ids.json dosyalarını dolu alır. Delta sync bu noktadan devam eder — kendi ortamınızda full sync çalıştırmanız gerekmez. Docker imajı da bu veriyi içinde gömülü taşır (haftalık tazelenir).

Mimari

  • GitHub Actions (Sync delta, hourly) → scripts/sync.py --mode delta
  • SGB API: siberguvenlik.gov.tr/api/address/index (newest-first, monoton artan ID)
  • Çıktı: docs/ altına düz metin dosyalar → GitHub Pages → Fastly CDN
  • State: state/seen_ids.json içinde her tür için max_id tutulur

CLI ile lokal çalıştırma

git clone https://github.com/bilsectr/sgb-api-bridge
cd sgb-api-bridge
pip install requests

python scripts/sync.py --mode delta    # ~1-3 dk, repo'daki veriden devam eder
python scripts/sync.py --mode loop     # docker icin: delta'yi surekli tetikler
python scripts/sync.py --mode full     # SADECE sifirdan tam re-sync gerekirse
